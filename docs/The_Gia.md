## 1. What Gia Is

Gia handles the complete journey of a lead — from the moment a form is submitted on a Meta ad, Google campaign, or website, through ingestion, validation, assignment, conversation, and resolution.

It is domain-scoped. An agent in `onboarding` sees only onboarding leads. A manager in `onboarding` sees all onboarding leads. No agent ever sees a lead outside their domain unless a cross-domain grant exists.

Gia does not replace WhatsApp or phone calls. It records them. The agent calls outside the system and comes back to Gia to log what happened.

## 2. Data Architecture

### Table: `leads`

```sql
-- Identity
id                  uuid          PRIMARY KEY DEFAULT gen_random_uuid()
slug                text          UNIQUE        -- human-readable URL key (priya-sharma-9182)
                                               -- immutable after insert. Generated by trigger.
first_name          text          NOT NULL
last_name           text
email               text
phone               text          -- stored as E.164 always

domain              app_domain    NOT NULL  -- onboarding | shop | legacy | house | b2b | finance | marketing | tech | concierge

-- Assignment
assigned_to         uuid          REFERENCES profiles(id)
assigned_at         timestamptz

-- Status & Intent
status              text          NOT NULL DEFAULT 'new'
                                  -- new | touched | in_discussion | won
                                  -- nurturing | lost | junk
lead_intent         text          -- hot | cold

-- Campaign & UTM
campaign_id         text
ad_name             text
platform            text          -- meta | google | website | whatsapp
utm_source          text
utm_medium          text
utm_campaign        text
utm_content         text

-- Raw form data
form_data           jsonb         -- original form submission, immutable after insert

-- Call tracking
call_count          integer       NOT NULL DEFAULT 0
last_call_outcome   text          -- rnr | switched_off | wrong_number | conversing | other

-- Deal (set on Won)
deal_type           text          -- membership | retail
deal_duration       text          -- 3_months | 6_months | 1_year (membership only)
deal_amount         numeric(12,2)

-- Agent workspace
private_scratchpad  text          -- only visible to assigned agent + admin + founder

-- Personal enrichment (JSONB, merged — never overwritten)
personal_details    jsonb         -- company, occupation, interests, city, notes

-- SLA tracking
status_changed_at   timestamptz   -- set on every status transition
last_activity_at    timestamptz   -- set on any call note or meaningful activity

-- Dedup
previous_lead_id    uuid          REFERENCES leads(id)  -- FK when terminal lead resubmits

-- Timestamps
created_at          timestamptz   NOT NULL DEFAULT now()
updated_at          timestamptz   NOT NULL DEFAULT now()
archived_at         timestamptz   -- soft delete only. Never hard delete.
```

**Key indexes:**

```sql
CREATE UNIQUE INDEX idx_leads_slug            ON leads(slug)              WHERE slug IS NOT NULL;
CREATE INDEX idx_leads_domain_status          ON leads(domain, status)    WHERE archived_at IS NULL;
CREATE INDEX idx_leads_assigned_to            ON leads(assigned_to)       WHERE archived_at IS NULL;
CREATE INDEX idx_leads_created_at             ON leads(created_at DESC)   WHERE archived_at IS NULL;
CREATE INDEX idx_leads_phone                  ON leads(phone);
CREATE INDEX idx_leads_phone_text             ON leads(phone text_pattern_ops) WHERE archived_at IS NULL;
CREATE INDEX idx_leads_utm_source             ON leads(utm_source)        WHERE archived_at IS NULL;
CREATE INDEX idx_leads_utm_campaign           ON leads(utm_campaign)      WHERE archived_at IS NULL;
CREATE INDEX idx_leads_last_call_outcome      ON leads(last_call_outcome) WHERE archived_at IS NULL;
```

**Slug format:** `priya-sharma-9182` — lowercase name parts + last 4 digits of E.164 phone, non-alphanumeric chars stripped. Generated by `trg_lead_slug` trigger on INSERT. **Immutable after creation** — phone or name changes never update the slug.

**RLS:**

- Agents see only their own leads (`assigned_to = auth.uid()`)
- Managers see all leads in their domain
- Admin/Founder see all leads in all domains

Append-only. Notes are never edited or deleted.

### Table: `lead_raw_payloads`

Immutable JSONB log of every inbound webhook payload.

```sql
id               uuid          PRIMARY KEY DEFAULT gen_random_uuid()
source           text          NOT NULL    -- meta | google | website
payload          jsonb         NOT NULL
lead_id          uuid          REFERENCES leads(id)  -- backfilled after lead insert
ingestion_error  text                                 -- non-null when ingestion failed
created_at       timestamptz   NOT NULL DEFAULT now()
```

Logged as step 1 of `ingestLead()` — before any extraction or validation. Logging failure is non-fatal and never blocks a lead. Admin/Founder SELECT only. No UPDATE or DELETE ever.

### Table: `agent_routing_config`

One row per agent. Controls round-robin eligibility.

```sql
id            uuid    PRIMARY KEY DEFAULT gen_random_uuid()
agent_id      uuid    NOT NULL REFERENCES profiles(id) UNIQUE
is_active     boolean NOT NULL DEFAULT true
shift_start   time                -- optional: 09:00
shift_end     time                -- optional: 18:00
updated_at    timestamptz NOT NULL DEFAULT now()
```

**`is_active`** is the holiday switch. Set to `false` → agent removed from assignment pool instantly.

### Table: `tasks` + `task_gia_meta`

See `docs/The_Blueprint.md` §Tasks for the full OS Tasks schema. For Gia, the relevant link is `task_gia_meta`:

## 3. Lead Ingestion — The Webhook

**Endpoint:** `POST /api/webhooks/leads?source=meta|google|website`

**Auth:** Bearer token via `PABBLY_WEBHOOK_SECRET`. Validated before the payload is read. Returns 401 immediately if invalid.

**Source:** Pabbly acts as middleware between Meta Ads, Google Ads, and website forms.

**Rate limiting:** 100 req/min per IP, applied at route level.

**Flow:**

```typescript
// 1. Rate-limit check
// 2. Log raw payload (non-fatal — never blocks lead)
// 3. Validate Bearer token → 401 if invalid
// 4. Source adapter: meta | google | website → NormalizedLeadPayload
// 5. ingestLead(normalizedPayload) →
//    a. Zod validation on normalized payload
//    b. Domain resolution (explicit domain > campaign prefix > default)
//    c. Phone dedup via get_active_lead_by_phone()
//       → Active dup: log duplicate_submission activity, return existing leadId
//       → Terminal dup: insert new lead with previous_lead_id FK
//    d. INSERT into leads (status = 'new', status_changed_at = now())
//    e. LOG lead_created activity
//    f. Backfill raw_payload_id onto lead_raw_payloads.lead_id
//    g. Assign agent via get_next_round_robin_agent() RPC
//    h. LOG agent_assigned activity (only if agent found)
//    i. Schedule SLA-01 timers (fire-and-forget)
//    j. Notify agent via WhatsApp (fire-and-forget)
// 5. Return 201 { leadId }
```

### Source Adapters — `src/lib/leads/adapters.ts`

Three adapters handle platform-specific payload shapes:

| Adapter | Priority order |
|---------|---------------|
| `adaptMeta` | (1) Meta native `field_data: [{name, values}]` → (2) Pabbly `raw_meta_fields: [{name, values}]` → (3) flat top-level keys |
| `adaptGoogle` | Google Ads form fields via `raw_google_fields` |
| `adaptWebsite` | Flat key-value form payload |

All adapters produce a typed `NormalizedLeadPayload`. Phone normalisation is wrapped in try/catch — stores raw on failure (defensive fallback; never rejects a lead on invalid phone from webhook). `sanitizeText()` applied to all text fields.

## 4. Domain Resolution

The domain of a lead is resolved from the `utm_campaign` field using a prefix mapping. This happens in `lib/services/lead-ingestion.ts` before the DB insert.

```typescript
// lib/constants/campaign-domain-map.ts
export const CAMPAIGN_DOMAIN_MAP: Record<string, string> = {
  TG_Global: "onboarding",
  TG_Shop:   "shop",
  TG_Legacy: "legacy",
  TG_House:  "house",
  TG_B2B:    "b2b",
};

export const DEFAULT_LEAD_DOMAIN = 'onboarding';

export function resolveDomainFromCampaign(campaignName: string | null): string {
  if (!campaignName) return DEFAULT_LEAD_DOMAIN;
  const entry = Object.entries(CAMPAIGN_DOMAIN_MAP).find(([prefix]) =>
    campaignName.startsWith(prefix),
  );
  return entry ? entry[1] : DEFAULT_LEAD_DOMAIN;
}
```

**Resolution priority:** explicit `data.domain` on the normalized payload takes precedence over campaign prefix, which takes precedence over the default.

**Fallback:** `onboarding`. Logged to Sentry as a warning when no prefix matches.

## 5. Agent Assignment — Round Robin

**Rule:** Every lead is assigned to an agent in the same domain as the lead.

**Algorithm:** DB-level round-robin via `get_next_round_robin_agent()` SECURITY DEFINER function. Uses `SELECT FOR UPDATE SKIP LOCKED` — race-free under concurrent webhooks. O(agents), not O(leads).

```sql
-- Atomic round-robin function (migration 0007)
-- Selects the eligible agent who received a lead the longest ago.
-- Agents without a routing config row are treated as having assigned_at = NULL (go first).
```

**Eligibility pool:**

- `profiles.domain = lead.domain`
- `agent_routing_config.is_active = true`
- `profiles.role = 'agent'`

**Holiday switch:** `agent_routing_config.is_active = false` removes an agent from the pool instantly. Flip back to `true` to re-enter.

## 6. Lead Lifecycle & Status

### Journey Statuses

```text
new → touched → in_discussion → won
```

### Resolution Statuses

```text
nurturing | lost | junk
```

### Status Definitions

| Status          | Meaning                              | Auto-action                                  |
| --------------- | ------------------------------------ | -------------------------------------------- |
| `new`           | Lead arrived, not yet called         | —                                            |
| `touched`       | First call attempt made              | Auto-set on first call log                   |
| `in_discussion` | Active conversation in progress      | —                                            |
| `won`           | Converted to client                  | Records deal (type + duration + amount)      |
| `nurturing`     | Not ready now, follow up in 3 months | Auto-creates task (due: now + 3 months)      |
| `lost`          | Will not convert                     | Requires reason                              |
| `junk`          | Invalid lead (wrong number, spam)    | Requires reason                              |

### Status Transitions — Atomic RPC

Every transition is handled by the `update_lead_status` RPC (migration 0031) called from `lib/actions/leads.ts`. The RPC runs in a single transaction:

1. Fetch current lead state
2. Early-return if status unchanged
3. `UPDATE leads SET status, status_changed_at, last_activity_at`
4. `INSERT lead_activities` (status_changed)
5. If `nurturing` → INSERT nurturing task + `task_gia_meta`

**Won side-effects (action layer, outside RPC):**

- Records deal fields (`deal_type`, `deal_duration`, `deal_amount`)
- Notifies managers via in-app notification + WhatsApp

**Lost/Junk:** require a `reason` field — stored in `lead_activities.details`.

**Revive:** `junk → in_discussion` is supported via `StatusActionPanel`. All history preserved. SLA timers reschedule because `in_discussion` is not a terminal SLA status.

Plain note composer (no call outcome, no `call_count` increment). Calls `addLeadNote` → `add_lead_plain_note` RPC.

**`LeadNotesSection`**
Chronological timeline of all `lead_notes` rows. Notes from `CalledModal` show the call outcome badge alongside the content.

**`LeadJourneyTimeline`**
Visual path: new → touched → in_discussion → terminal outcome. Dwell time shown per stage. Terminal badge in header. Built from `lead_activities` timestamps.

**`LeadDossierTasksAsync`**
Async server component. Shows next pending task. Overdue state highlighted.

**`DynamicFormResponses`**
Renders `leads.form_data` JSONB as field → value pairs.

**`LeadActivityLog`**
Audit log of all `lead_activities` rows with actor names.

## 8. Call Logging

Agent clicks **Called** → `CalledModal` opens.

### CalledModal Flow

1. Agent selects call outcome (required)
2. Agent types a note (required, min 1 char)
3. Agent submits → `addLeadCallNote` server action

### Call Outcomes

| Value          | Label              |
| -------------- | ------------------ |
| `rnr`          | Rang, no response  |
| `switched_off` | Phone switched off |
| `wrong_number` | Wrong number       |
| `conversing`   | Had a conversation |
| `other`        | Other              |

### Server Action: `addLeadCallNote` → RPC `add_lead_call_note`

Migration 0030. Single transaction wrapping all DB writes:

```typescript
// Action layer (before RPC):
// 1. Zod validation (first line)
// 2. Auth check — getCurrentProfile()
// 3. Access verification via RLS-bound client

// RPC (atomic):
// 1. Fetch current lead state
// 2. INSERT lead_notes
// 3. UPDATE leads: call_count++, last_call_outcome, last_activity_at
//    If status = 'new' → also update status = 'touched', status_changed_at
// 4. INSERT lead_activities (call_logged)
// 5. INSERT lead_activities (note_added)
// 6. If auto-advance: INSERT lead_activities (status_changed new→touched)
// 7. Return: { note_id, new_call_count, did_auto_advance, assigned_to, domain, old_status }

// Action layer (after RPC, using returned data):
// 8. Update last_activity_at for SLA refresh
// 9. Cancel + reschedule SLA timers based on new status
```

**Auto-advance rule:** First call on a `new` lead automatically sets status to `touched`. The only automatic status transition that does not require explicit agent action.

**Plain notes** (via `LeadNotesInput`, not `CalledModal`) use the `add_lead_plain_note` RPC (migration 0040). Plain notes do NOT increment `call_count` or update `last_call_outcome`.

## 9. Notes System

One type. One principle: notes are never edited or deleted.

### Call Notes (via CalledModal)

- Written via the `CalledModal` on the dossier
- Always have a `call_outcome` value
- Increment `call_count` + update `last_call_outcome` on the lead
- Trigger auto-advance `new → touched` on first call

### Plain Notes (via LeadNotesInput)

- Written via the notes UI on the dossier, not tied to a call
- `call_outcome = null` — no `call_count` increment, no outcome update
- Use case: adding context, follow-up notes, WhatsApp message summaries

### Display

Both types shown in `LeadNotesSection` as a chronological timeline. Call notes show an outcome badge. Plain notes show no badge.

**Visibility:** all notes visible to anyone with domain access to this lead.

**The scratchpad is not notes.** `leads.private_scratchpad` is a live working space — not logged, not append-only, cleared on reassignment.

## 10. Tasks on a Lead

Tasks on a lead come from four sources.

### Source 1 — Auto-created on Nurturing

When lead status transitions to `nurturing` (via RPC):

```text
task_type:   general_follow_up
status:      to_do
due_at:      now() + interval '3 months'
assigned_to: lead's current assigned agent (falls back to actor_id)
```

### Source 2 — Auto-created by SLA Engine

When an SLA breach fires for an agent rule, a task is created (if no open gia_followup task already exists):

```text
task_type:   general_follow_up   (task_category: gia_followup)
titles:
  SLA-01A: 'New lead untouched — follow up now'        (priority: urgent)
  SLA-02A: 'No update in 24 hours — follow up on lead' (priority: high)
  SLA-03A: 'Discussion stalled — re-engage lead'        (priority: high)
  SLA-04A: 'Less than 3 call attempts in 4 days'        (priority: high)
```

### Source 3 — Auto-created by retry scheduling

When an agent logs `rnr` or `switched_off`, they can schedule a retry from `CalledModal`:

```text
task_type:   call
due_at:      agent-selected date/time
assigned_to: lead's current assigned agent
```

### Source 4 — Manually created

From the dossier:

| `task_type`         | Description         |
| ------------------- | ------------------- |
| `call`              | Scheduled call      |
| `whatsapp_message`  | WhatsApp follow-up  |
| `email`             | Email follow-up     |
| `general_follow_up` | Any other follow-up |

Tasks displayed in `LeadDossierTasksAsync` — next due task only inline on the dossier. Full task list in the Tasks module.

## 11. Lead List & Access Control

**Route:** `app/(dashboard)/leads/page.tsx`
**Architecture:** Three-component Suspense split

```text
leads/page.tsx                        ← Server component (thin orchestrator)
  ├── <LeadsFilters />                ← Client, URL param sync only
  └── <Suspense fallback={skeleton}>
        <LeadsTableAsync />           ← Async server, fetches filtered leads
          └── <LeadsTable />          ← Display-only, column picker, pagination
```

### Access Rules

| Role      | Sees                                                     |
| --------- | -------------------------------------------------------- |
| `agent`   | Only leads assigned to them (`assigned_to = auth.uid()`) |
| `manager` | All leads in their domain                                |
| `admin`   | All leads in all domains                                 |
| `founder` | All leads in all domains                                 |

Role constraint enforced in `getLeadsByRole` **before** any filter is applied. A crafted `?agent_id=<other-uuid>` URL never exposes another agent's leads.

### Filters Available

| Filter         | Options                                                       |
| -------------- | ------------------------------------------------------------- |
| Search         | Server-side ilike on first_name, last_name, phone, email (500ms debounce) |
| Status         | new / touched / in_discussion / won / nurturing / lost / junk (multi) |
| Last outcome   | rnr / switched_off / wrong_number / conversing / other (multi) |
| Domain         | admin/founder only; selects from GIA_DOMAIN_FILTER_ITEMS     |
| Campaign       | single, from distinct utm_campaign values                     |
| Source         | meta / google / website / whatsapp (single)                  |
| Date range     | created_at between X and Y                                    |
| Assigned agent | manager/admin/founder only                                    |

**Four invariants (non-negotiable):**

1. `getLeadsByRole` returns `{ leads: LeadWithAssignee[], totalCount: number }` — never `Lead[]` alone
2. `totalCount` from `{ count: 'exact', head: false }` on the same query builder — never a second COUNT query
3. Every filter/search URL push deletes the `page` param (enforced in `buildParams()`)
4. `LeadsTable.tsx` contains zero filtering, searching, or sorting logic — display only

## 12. The Full Flow — End to End

```text
INGESTION
─────────
Webhook arrives (Meta / Google / Website via Pabbly)
→ Rate-limit check (100 req/min per IP)
→ Log raw payload to lead_raw_payloads (non-fatal)
→ Validate Bearer token (reject 401 if invalid)
→ Source adapter normalizes payload
→ ingestLead():
    → Zod validation on normalized payload
    → Domain resolution (explicit > campaign prefix > default 'onboarding')
    → Phone dedup:
        active lead found → duplicate_submission activity, return existing leadId
        terminal lead found → new lead with previous_lead_id FK
    → INSERT leads (status = 'new', slug auto-generated by trigger)
    → LOG lead_created activity
    → Atomic round-robin via get_next_round_robin_agent() DB function
    → UPDATE leads SET assigned_to, assigned_at
    → LOG agent_assigned activity (if agent found)
    → Schedule SLA-01 timers via Trigger.dev
    → Notify agent via WhatsApp
→ Return 201 { leadId }

FIRST CONTACT
─────────────
Agent receives WhatsApp notification
→ Opens lead dossier (slug URL: priya-sharma-9182)
→ Calls the lead (outside Eia)
→ Clicks Called → CalledModal opens
→ Selects outcome + writes note (both required)
→ Submits → add_lead_call_note RPC (atomic):
     INSERT lead_notes
     UPDATE call_count++, last_call_outcome, last_activity_at
     if status = 'new' → UPDATE status = 'touched', status_changed_at
     INSERT lead_activities (call_logged + note_added)
     if auto-advance → INSERT lead_activities (status_changed new→touched)
→ Action layer: reschedule SLA timers

PROGRESSION
───────────
Agent updates status as lead warms:
→ touched → in_discussion
   Each transition: update_lead_status RPC → UPDATE leads.status + LOG status_changed

RESOLUTION
──────────
Won:
→ WonDealModal: deal_type + deal_duration (membership) + deal_amount
→ recordDeal action → update deal fields + call updateLeadStatus('won')
→ update_lead_status RPC: UPDATE leads.status + LOG status_changed
→ Cancel SLA timers
→ Notify domain managers

Nurturing:
→ update_lead_status RPC: UPDATE leads.status + LOG status_changed
   + Auto-create nurturing task (due: +3 months) within same transaction
→ Cancel + reschedule SLA-04 timers

Lost:
→ ReasonModal: reason required
→ update_lead_status RPC: UPDATE leads.status + LOG status_changed with reason
→ Cancel SLA timers

Junk:
→ ReasonModal: reason required
→ update_lead_status RPC: UPDATE leads.status + LOG status_changed with reason
→ Cancel SLA timers

Revive (junk → in_discussion):
→ ConfirmModal
→ update_lead_status RPC: UPDATE leads.status = 'in_discussion'
→ SLA-03 timers reschedule (in_discussion is not terminal)

AUDIT TRAIL
───────────
Everything above leaves an immutable row in lead_activities.
No UPDATE or DELETE ever runs on lead_activities or lead_notes.
The full history of every lead is always recoverable.
```

## 13. WhatsApp

WhatsApp is the second ingestion channel into Gia — alongside the webhook from Meta/Google/website forms.

Any message that arrives on the company WhatsApp Business number from an unknown contact creates a lead automatically. The agent works that lead the same way as any other — through the dossier, with the same status lifecycle, tasks, and notes. The only difference is the origin.

The system has two built parts:

1. **Ingestion** — inbound messages from unknown numbers create leads via the Meta WhatsApp Business API
2. **The WhatsApp Page** — a full in-app messaging interface where agents read and reply to all WhatsApp conversations without leaving Eia

A third part is planned but not built yet:

3. **The AI Chatbot** — a Claude-powered assistant that engages the lead automatically

---

### 13.1 How Ingestion Works

**Endpoint:** `POST /api/webhooks/whatsapp`

**Auth:** Meta webhook signature (`X-Hub-Signature-256` header). Requests without a valid signature are rejected with 401.

**Flow on inbound message:**

```text
Message arrives on company number
→ Verify Meta signature (reject 401 if invalid)
→ Extract sender phone (wa_id) + message content
→ Normalise phone to E.164
→ Check: does a lead already exist with this phone?

  IF YES → attach message to existing lead conversation
            no new lead created

  IF NO  → INSERT into leads:
              first_name: wa_id (temporary, until agent updates)
              phone:      normalised E.164
              domain:     'onboarding' (default — agent can reassign)
              status:     'new'
              platform:   'whatsapp'
            LOG lead_created activity
            Assign agent via round-robin
            Notify assigned agent
```

**Duplicate guard:** Phone is the dedup key. If a lead with that E.164 number already exists in any status (including won, lost, junk), the message threads into that lead's conversation.

**Domain assignment:** WhatsApp leads default to `onboarding`. Agent or manager can reassign from the dossier.

### 13.4 Webhook: Inbound Messages & Delivery Receipts

**Type 1 — New message (inbound):**

```text
1. Verify signature
2. Extract wa_id, message content, type, wa_message_id
3. Check whatsapp_messages for wa_message_id → skip if exists (dedup)
4. Resolve lead by phone → found: get/create conversation row | not found: INSERT lead + conversation
5. INSERT whatsapp_messages (direction: inbound, sender_type: lead)
6. UPDATE whatsapp_conversations SET last_message_at = now()
7. Push Realtime event to agent
8. Return 200 immediately
```

**Type 2 — Status update (delivery receipt):**

```text
UPDATE whatsapp_messages
SET status = $status, status_at = now()
WHERE wa_message_id = $wa_message_id
```

**Always return 200 to Meta.** If Meta receives anything else, it retries.

## 14. SLA Engine

The Gia SLA Engine enforces response-time commitments using event-driven Trigger.dev delayed jobs. No polling. No cron scanners.

### Business Hours

```text
Timezone:   Asia/Kolkata (IST, UTC+5:30)
Start:      09:00 IST (inclusive)
End:        19:00 IST (exclusive)
Off days:   Sunday — Saturday is a business day
```

Source of truth: `src/lib/constants/sla.ts` (`BUSINESS_HOURS` constant).

### SLA Rules

| Code    | Status Trigger  | Threshold           | Recipient | Auto-task? |
| ------- | --------------- | ------------------- | --------- | ---------- |
| SLA-01A | `new`           | 15 business minutes | Agent     | Yes        |
| SLA-01B | `new`           | 30 business minutes | Manager   | No         |
| SLA-02A | `touched`       | 24 business hours   | Agent     | Yes        |
| SLA-02B | `touched`       | 36 business hours   | Manager   | No         |
| SLA-03A | `in_discussion` | 24 business hours   | Agent     | Yes        |
| SLA-03B | `in_discussion` | 36 business hours   | Manager   | No         |
| SLA-04A | `nurturing`     | 4 business days     | Agent     | Yes        |
| SLA-04B | `nurturing`     | 4 business days     | Manager   | No         |

"A" rules notify the assigned agent. "B" rules escalate to all active managers/admins/founders in the lead's domain.

### Escalation Chain

1. SLA-01A fires → agent notified + auto-task created (if no open gia_followup task exists)
2. SLA-01B fires (15 min later) → all domain managers notified
3. Lead advances to `touched` → SLA-01 cancelled, SLA-02 timers start fresh
4. On each call note: SLA-02/03 timers reset; SLA-01 never reset by activity

### Auto-task Guard

Before creating an auto-task, `fireSlaBreachHandler` calls `getOpenGiaFollowupTask(leadId, assignedTo)`. If an open task already exists, no new task is created. Prevents duplicate tasks from stale-fire retries.

### Double-scheduling Guard

`scheduleLeadSlasTask` passes `idempotencyKey: 'lead-sla-${leadId}-${ruleCode}'`. Trigger.dev v3 deduplicates by key for all non-terminal run states including DELAYED. Double-scheduling is structurally impossible.

### Stale-fire Guard

`fireLeadSlaTask` re-reads the lead status from DB on every execution. If status no longer matches `statusTrigger`, logs `sla_breach` activity with `outcome: 'stale_fire'` and exits cleanly.

### Terminal Statuses

`won`, `lost`, `junk` → cancel all SLA timers, schedule none.

## 16. Decision Log

| Date | Decision | Chosen | Why |
| --- | --- | --- | --- |
| — | Notes: private vs common type | All notes in `lead_notes` visible to anyone with domain access. Private thoughts belong in `leads.private_scratchpad` only. | Scratchpad covers the private use case. Two visibility tiers adds RLS complexity and a choice agents don't need on every note. |
| — | Scratchpad on reassignment | Clear `private_scratchpad` on reassign | Scratchpad is private to the assigned agent. If context needs to transfer, outgoing agent writes a note before reassignment. |
| — | Task architecture | Core `tasks` table + per-module extension tables (`task_gia_meta`, etc.) | A single god table with nullable module-specific columns is unmaintainable at scale. |
| 2026-05-28 | Atomic round-robin | `get_next_round_robin_agent()` SECURITY DEFINER with `SELECT FOR UPDATE SKIP LOCKED` | Two concurrent webhook calls cannot pick the same agent. O(agents) not O(leads). |
| 2026-05-28 | Lead dedup strategy | Phone as dedup key. Active lead → duplicate_submission activity. Terminal lead → new lead with previous_lead_id FK. | A phone number in an active conversation should never fork into two dossiers. |
| 2026-05-28 | Campaign analytics in leads-service | `getCampaignMetrics` and `getCampaignDetailMetrics` in `leads-service.ts` | Campaign data derives directly from the `leads` table. No separate module concern. |
| 2026-05-29 | addLeadCallNote and updateLeadStatus as RPCs | Collapsed 9 and 5 sequential DB awaits into single SECURITY DEFINER RPC transactions (migrations 0030, 0031). SLA side-effects remain in action layer. | Sequential awaits leave partial state visible between steps. Single transaction is atomic. SLA involves Trigger.dev calls that cannot roll back. |
| 2026-05-29 | SLA timer table writable by service role only | No INSERT/UPDATE/DELETE RLS for regular users — service role only via Trigger.dev. | SLA timers are infrastructure state, not user-facing data. |
| 2026-05-29 | add_lead_plain_note RPC separate from add_lead_call_note | Plain notes (no outcome, no call_count increment) get their own RPC (migration 0040). | Mixing the two conflates a note-taking action with a call-logging action. The call_count / last_call_outcome fields must only move when an actual call happened. |
| 2026-05-30 | Lead slug as URL identifier | `priya-sharma-9182` format — immutable after insert, generated by DB trigger. | Human-readable URLs survive agent handoffs. Immutable avoids broken bookmarks when name/phone changes. |
| 2026-05-30 | DEFAULT_LEAD_DOMAIN = 'onboarding' | Migrated from 'concierge' (migration 0041) | 'concierge' is a staff domain, not a lead-receiving domain. All unknown UTM prefixes should land in onboarding. |
| 2026-05-31 | Deal fields on leads | `deal_amount`, `deal_type`, `deal_duration` on `leads` table (migration 0049). Membership requires duration; retail requires amount only. | Single record per lead; no separate deals table needed at current scale. |
