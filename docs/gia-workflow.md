# Full Lead Pipeline Audit

## Stage 1 — Webhook entry `POST /api/webhooks/leads`

**Status: Clean.**

- Rate limiting (100 req/min per IP) before anything else — correct.
- Raw payload logged to `lead_raw_payloads` as step 1 (non-fatal — never blocks a lead) — correct.
- Bearer token validated before body is read — correct (Rule S-12).
- Source query param (`meta | google | website`) defaults to `'website'` — correct.
- Returns 201 with `leadId` on success — correct.

## Stage 2 — Source adapters `src/lib/leads/adapters.ts`

**Status: Clean.**

- Three adapters (`adaptMeta`, `adaptGoogle`, `adaptWebsite`) each produce a typed `NormalizedLeadPayload`.
- `adaptMeta` handles three payload shapes in priority order: (1) Meta native `field_data: [{name, values}]`, (2) Pabbly `raw_meta_fields: [{name, values}]`, (3) flat top-level keys — multi-key fallback for phone/email/ad fields; all non-standard keys captured into `form_data`.
- Phone normalisation wrapped in try/catch — stores raw on failure rather than rejecting (defensive fallback; webhooks must never silently drop a lead).
- `sanitizeText()` applied to all text fields — correct.
- `splitName()` handles full_name → first/last correctly.

## Stage 3 — Lead ingestion `src/lib/services/lead-ingestion.ts`

**Status: Clean.**

- Raw payload logged as step 1 (non-fatal).
- Zod validation on normalized payload.
- Domain resolution: explicit `data.domain` takes precedence, then campaign prefix mapping, then `DEFAULT_LEAD_DOMAIN = 'onboarding'` (migrated from `'concierge'` in migration 0041).
- Dedup by phone: `get_active_lead_by_phone()` RPC.
  - Active lead → `duplicate_submission` activity logged, return existing `leadId`.
  - Terminal lead → new lead inserted with `previous_lead_id` FK.
- Uses `createAdminClient()` for DB insert and activity logs — correct (webhook has no session).
- `status_changed_at` and `last_activity_at` set on insert (migration 0027).
- `assigned_at` set to `null` when no agent is found — correct.
- Logs `lead_created` activity unconditionally, `agent_assigned` only when an agent is found.

## Stage 6 — Server actions `src/lib/actions/leads.ts`

**Status: Clean.**

All actions start with Zod validation (Rule S-01). Auth check via `getCurrentProfile()` before any DB work (Rule A-09). RLS-bound `createClient()` for access verification; `createAdminClient()` for mutations.

| Action | What it does | Atomicity |
| ------ | ----------- | --------- |
| `addLeadCallNote` | Log call + outcome + note | RPC `add_lead_call_note` (migration 0030) — 9 sequential awaits collapsed to 1 transaction. SLA side-effects in action layer. |
| `addLeadNote` | Plain note (no call_count increment) | RPC `add_lead_plain_note` (migration 0040) — does NOT update `call_count` or `last_call_outcome`. |
| `updateLeadStatus` | Status transition | RPC `update_lead_status` (migration 0031) — 5 sequential awaits collapsed to 1 transaction. Won notifications and SLA side-effects in action layer. |
| `assignLead` | Reassign lead | Pre-update `SELECT status, domain` → UPDATE. Clears `private_scratchpad` on reassign. Schedules SLA-01 timers. |
| `updateLeadInfo` | Edit contact fields | UPDATE first_name, last_name, phone, email. |
| `updatePersonalDetails` | Enrich lead JSONB | Zod → auth → access check → JSONB merge (preserves prior keys, strips empty strings). |
| `updateScratchpad` | Save private notes | Debounced 1s in component, direct UPDATE server-side. |
| `createManualLead` | Manual lead creation | Dedup check → INSERT → round-robin assign → schedule SLA. Returns `{ duplicate: true, leadId }` on dup. |
| `recordDeal` | Record deal on Win | Updates `deal_type`, `deal_duration`, `deal_amount` + calls `updateLeadStatus('won')`. |
| `listAgentsForDomain` | Fetch agents for dropdowns | Read-only. Used by `AddLeadModal` and dossier reassignment. |

All actions return `{ data, error }`, never throw, never void — Rule Q-03 satisfied.

## Stage 11 — SLA Engine hook points

**Status: Clean.**

- `assignLead` + `createManualLead`: after assignment, updates `status_changed_at` + `last_activity_at`, then schedules SLA-01 timers (fire-and-forget, non-blocking).
- `updateLeadStatus`: after status write, cancels existing timers. Terminal status (`won`, `lost`, `junk`) → cancel only. Non-terminal → cancel then reschedule for new status.
- `addLeadCallNote`: after note write, updates `last_activity_at`. If auto-advanced `new → touched` → full SLA reset. Otherwise → refreshes SLA-02/03 timers only (SLA-01 never reset by activity).
- `addLeadNote` (plain): updates `last_activity_at` only. Does NOT affect SLA timers — plain notes are not call events.
- Stale-fire guard: `fireLeadSlaTask` re-reads lead status from DB on execution; exits cleanly with `outcome: 'stale_fire'` if status has changed.
- Idempotency key: `lead-sla-${leadId}-${ruleCode}` — Trigger.dev deduplicates DELAYED runs. Double-scheduling is structurally impossible.

## Stage 12 — WhatsApp ingestion

**Status: Clean.**

- Meta signature verification (`X-Hub-Signature-256`) before payload read.
- `wa_message_id` dedup check before any insert — prevents duplicate rows from Meta retries.
- Phone dedup: same `get_active_lead_by_phone()` RPC used by the main webhook.
- `whatsapp_conversations` one-row-per-lead constraint enforced by UNIQUE on `lead_id`.
- Always returns 200 to Meta — process errors logged to Sentry, never surfaced as HTTP errors.
- Delivery receipts (status updates) are the only mutation allowed on `whatsapp_messages` rows.
- Realtime push to agent on new inbound message.
