# graphify-out/ — how to navigate this knowledge graph

This directory holds the Serene knowledge graph and its derived views. Two layers, different jobs:

## 1. Curated navigation layer — **start here** → [`wiki/index.md`](wiki/index.md)

Hand-authored, kept current with the *new* feature names (Elaya, Lead Revival, Web Push, voice
dictation, performance drill modals, SLA engine, …). One page per real subsystem, each with the same
shape: **Purpose → Entry points → Data flow → Canonical helpers → Tables → Invariants/Gotchas → File map.**
Plus [`wiki/_conventions.md`](wiki/_conventions.md) — the cross-cutting laws that bite (the `after()`
rule, dual-key cache, Zod-first actions, the PII gateway, IST math), indexed by where they apply.

This is the layer to read for *"where does X live and what must I not break."*

## 2. Raw graph topology — for broad architecture review only

- `graph.json` — the full node/edge graph (6,253 nodes · 11,431 edges). Query it, don't read it:
  - `graphify query "<question>"` — BFS scoped subgraph (much smaller than the report).
  - `graphify path "<A>" "<B>"` — shortest path between two nodes.
  - `graphify explain "<concept>"` — a node + its neighbours.
- `GRAPH_REPORT.md` — auto-generated cluster report. **Regenerated by `graphify` — do not hand-edit**
  (edits are overwritten). Many community names are machine placeholders ("Dev Guide · Cluster 33",
  "Community 79"); for human/agent navigation prefer the `wiki/` above.
- `manifest.json`, `.graphify_*`, `cache/`, dated `YYYY-MM-DD/` snapshots — graphify internals.

## Keeping it fresh

- After code changes: `graphify update .` (AST-only, no API cost) — refreshes `graph.json` topology.
- After a subsystem changes *shape* (new service file, new table, new invariant): update the relevant
  `wiki/*.md` page by hand. The wiki is curated, not generated — that's the point.
